Page Header

The title block at the top of a view: it names where the user is, gives context, and holds the page's primary actions. For headings that introduce a block within a page, use the Section Header instead.

Anatomy

One composable block: a title with optional context above and below it, optional actions on the right, and a divider that separates it from the page content.

Best practices

It's one composable block — fill only the parts a view needs.

→
One per view. A single Page Header sits at the top of a view and is the page's h1. To head a block within a page, use a Section Header.
→
Name the view specifically. The title should say exactly what this view is; put the broader context — a project or pulse — in the eyebrow above it.
→
The subtitle is flexible. The line under the title can be a description, a date range, a live status, ownership or a scope — alone or combined. It should add context, not repeat the title.
→
Keep actions page-level. The right side holds the one action — or search & sort — for the whole view, never row- or item-level controls.
→
Tabs only for real sub-sections. When a view splits into sub-views the tab bar sits just below the title; see Tabs.
→
The divider runs full-bleed. The bottom border spans the full width of the view, edge to edge, while the title, subtitle and actions keep the container's side padding.

Accessibility

The header anchors the page for everyone, so its structure has to be sound for assistive tech.

→
The title is the page's h1. There should be exactly one per view, and the rest of the page's headings nest beneath it.
→
The eyebrow is context, not a heading. Don't mark it up as its own heading level — keep it part of the header block so the heading outline stays clean.
→
Name icon-only actions. A settings or options icon button needs an accessible name ("Settings"), since there's no visible label.
→
Status isn't colour-only. "Running" must read as text, not just a coloured dot, so the state is clear without colour perception.
→
Tabs use the tab pattern. When the header carries tabs, follow the Tabs roles and keyboard behaviour; the divider is decorative.

Accessibility

Get the document structure right in markup.

→
Title is the <h1>. Render the title as the single page h1; don't promote the eyebrow to a heading.
→
Action names. Icon-only buttons need aria-label; link-outs that open a new tab should say so.
→
Tabs semantics. Use the Tabs component's roles (role="tablist" / tab / tabpanel) rather than re-implementing them.
→
Status text. Pair the status dot with a text label so state never relies on colour alone.
